Learned Indexes for Filesystem Metadata: A From Scratch Implementation and Benchmark Study

July 23, 2026

Managing filesystem metadata, such as inode tables and directory entries, is fundamentally a key value indexing problem. Systems like TableFS and BetrFS have addressed this by utilizing database storage engines to improve speed. Alternatively, learned indexes offer a novel approach by replacing large tree data structures with small machine learning models that approximate the storage locations of keys. This project explores that intersection.

I present an in memory implementation of three index structures built entirely from scratch. These include a sorted array, a block based B+tree, and a two stage learned index equipped with a delta buffer for insertions. The goal was to rigorously measure lookup latency, memory overhead, and insertion cost across various key distributions scaling up to five million keys.

At five million keys, the learned index performed lookups nearly four times faster than the array or B+tree alternatives. Furthermore, it required almost seven hundred times less memory for the index itself. However, handling insertions introduced significant complexity and cost. The following sections detail the implementation, the testing methodology, and the limits of this design.

Introduction

Two primary concepts motivated this study. First, standard filesystems struggle with metadata heavy workloads involving millions of small files. Projects like TableFS resolve this by implementing database storage engines in place of traditional metadata structures.

Second, learned indexes demonstrate that classic indexes essentially approximate a function mapping a key to its physical location. Replacing a large tree with a small mathematical model allows for faster lookups and drastically reduced memory usage, provided there is a reliable method for local error correction.

This project applies the learned index concept to filesystem metadata lookups. I conducted a single node, in memory study focused exclusively on the index layer. The study does not evaluate a complete filesystem, distributed systems, or actual physical storage drives.

Related Work

DBMS backed filesystem metadata. TableFS stores file metadata within an LSM tree rather than typical inode blocks. BetrFS replaces the entire storage structure with B epsilon trees. Both designs seek to achieve high performance for random writes and sequential scans.

Learned indexes. Kraska and colleagues introduced the Recursive Model Index. This design utilizes a cascade of models where each step selects a more specific model, concluding with a bounded local search. Subsequent research expanded on this concept. FITing Tree and the PGM index provided mathematical worst case error bounds. RadixSpline optimized the build process for speed. ALEX introduced gapped arrays to handle insertions efficiently. The delta buffer design in this study draws heavy inspiration from ALEX.

Learned indexes for metadata. SwiftKV proposed integrating LSM trees with learned indexes to resolve read bottlenecks in distributed filesystems. A recent benchmark study examined how learned indexes compete for memory against caches and Bloom filters. Other implementations have adapted learned indexes for flash translation layers. Building on this existing research, my objective was to produce an independent reproduction in a simplified setting.

System Design

All three implemented structures serve the same purpose. They accept a 64 bit key and return its logical position or report its absence.

Sorted array with binary search (baseline)

This serves as the minimal reference point. Lookups utilize a standard binary search over a flat sorted array. Since the array itself acts as the index, it requires zero additional memory. Insertions, however, require shifting every subsequent key to the right. This represents the exact performance issue that database indexes were created to solve.

Block based B+tree style index

I constructed a simplified two level index resembling SSTable index blocks. It features fixed capacity sorted leaf blocks of 128 keys each and a sparse root array tracking the maximum key per leaf. Lookups perform a binary search on the root array followed by a binary search within the target leaf. Insertions require shifting items locally within a leaf and splitting the leaf upon overflow. While not a fully recursive B+tree, it perfectly mirrors the performance characteristics of production storage engines.

Learned index (two stage RMI) with a delta buffer

Stage zero consists of a single linear model that routes a key to one of several models in Stage one. Stage one contains multiple independent linear regressions. Each regression covers a specific subset of keys and records its maximum prediction error during training. A lookup evaluates Stage zero, queries the appropriate Stage one model, and performs a binary search bounded by the recorded error margin.

Because learned indexes are trained on static data, they require modifications to support insertions. I implemented a delta buffer consisting of a small sorted array. Lookups check the learned index first and consult the delta buffer if necessary. All new insertions are directed to the delta buffer. When the buffer reaches capacity, its contents merge into the main array and the machine learning model retrains completely. Figure 1 illustrates this data flow.

Methodology

Implementation. I developed all three structures and the benchmarking suite in C. The code relies solely on standard math libraries and was compiled with basic optimizations.

Key distributions. I generated three distinct sets of synthetic 64 bit keys to simulate typical filesystem metadata.

  • Sequential keys consisted of monotonically increasing numbers to represent append only logs.
  • Uniform keys were generated randomly and sorted, serving as an ideal proxy for hashed paths.
  • Lognormal keys simulated real world skew where files cluster based on timestamps or sizes. Due to the high clustering in lognormal data, tests requiring five million unique keys reached their limit at roughly 2.86 million distinct values.

Lookup benchmark. I populated the indexes and executed 200,000 random point lookups per configuration. The test recorded the average nanoseconds required to locate a key across all three distributions and sizes ranging up to five million keys.

Insert benchmark. I partitioned the keys, building the initial index with eighty percent of the data and streaming the remaining twenty percent as insertions. I carefully randomized the insertion order to avoid creating an artificially fast sequential append operation. I limited these tests to 200,000 keys because the baseline array performance degraded severely at larger sizes.

Hardware. All experiments ran on a standard Linux container utilizing a single thread. The results provide a direct comparison of the structures on identical hardware rather than absolute performance metrics for production environments.

Results

Lookup latency vs index size

The learned index consistently outperformed the baselines, with the performance gap widening as the dataset expanded. At 10,000 keys, the learned index was more than twice as fast as the array. At five million keys, the learned index required 123.2 nanoseconds per lookup, making it roughly 3.4 times faster than the array and 4 times faster than the B+tree. A well trained model successfully replaces logarithmic searching with fast constant time arithmetic.

In certain scenarios, the B+tree performed slightly slower than the basic array. This likely stems from my chosen leaf capacity of 128 keys causing cache misses, rather than an inherent flaw in B+trees. A highly optimized library implementation would likely close this performance gap.

Index memory overhead

The memory efficiency represents the most significant advantage in this study. The learned index requires only a few floating point numbers for its models, whereas the B+tree must store extensive arrays of leaf and root pointers. The learned index consistently utilized almost 688 times less memory across all tested sizes.

Insert cost

At 200,000 keys, the basic array required 19.1 microseconds per insertion. The B+tree processed insertions natively in 2.78 microseconds. The learned index averaged just 1.14 microseconds per insertion. This impressive average occurs because most insertions drop quickly into the delta buffer. However, this average conceals the massive computational penalty incurred when the buffer fills and forces a complete model retrain.

Lookup latency during sustained inserts

I expected latency to spike significantly as the delta buffer filled. Surprisingly, performance remained relatively stable. Lookups hovered near 117 nanoseconds throughout the insertion stream, compared to 250 nanoseconds for the B+tree. The delta buffer successfully absorbed new data without allowing the main model error rate to degrade unacceptably.

Discussion

The data clearly demonstrates why researchers favor learned indexes. For read heavy workloads with relatively static data, a small mathematical model easily outperforms classic tree structures in both speed and memory efficiency. The drastically smaller memory footprint allows systems to retain more of the index in the CPU cache.

However, writing data introduces significant complications. The delta buffer effectively managed the average insertion cost, but the implementation is notably more complex than a standard B+tree split. Lookups must check two separate structures, and the entire system must halt periodically to retrain the models. Safely managing this retraining process in a highly concurrent production environment would present a formidable engineering challenge.

Limitations and Threats to Validity

This study includes several explicit limitations.

  • In memory only. The system never interacted with physical storage drives or page caches. Real filesystem lookups are typically restricted by I/O limits, which this study does not measure.
  • Synthetic keys. The generated distributions serve as reasonable proxies but do not perfectly replicate real filesystem traces containing complex burst patterns.
  • Single threaded. The implementation lacks locks or mutexes. A production metadata layer must handle thousands of concurrent operations safely.
  • Simplified B+tree. The baseline B+tree implementation was relatively basic. A fully optimized library implementation would likely deliver superior performance.
  • Not a filesystem. This project solely evaluates a single index layer and does not represent a functional filesystem or provide crash recovery mechanisms.

Conclusion and Future Work

I built three distinct key to position indexes to evaluate their performance on filesystem style lookups. The two stage learned index performed lookups four times faster than a standard array or B+tree while using nearly 700 times less memory. This performance advantage came at the expense of a vastly more complex insertion process. These findings strongly validate the established academic literature regarding learned indexes.

Future work should involve backing the index with mapped or direct I/O storage. Replacing the delta buffer with a true gapped array design would enable genuine in place insertions. Implementing robust concurrency control and testing the system against authentic filesystem traces remain critical next steps.
