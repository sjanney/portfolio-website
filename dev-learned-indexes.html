Learned Indexes for Filesystem Metadata: A From Scratch Implementation and Benchmark Study

July 23, 2026

If you've taken an intro systems course, you've probably built a hash table or a B-tree and been told "this is how databases index things." What you might not have spent much time on is filesystem metadata — the inode tables and directory entries that tell your OS where a file actually lives on disk. It turns out this is the exact same problem in disguise: given a key (a filename, an inode number, a path), find the location of the data it points to. It's a key-value indexing problem wearing a filesystem costume.

Traditional filesystems handle this with tree structures — B-trees, B+trees, and their many variants. They work, but they weren't really designed with today's workloads in mind: filesystems full of millions of small files, like you'd see in object storage, container images, or ML training datasets with millions of tiny sample files. Projects like TableFS and BetrFS noticed this mismatch and responded by swapping in full database storage engines (like LSM-trees) in place of classic inode structures, trading some complexity for much better performance on metadata-heavy workloads.

Around the same time, a different idea started circulating in the database systems community: learned indexes. The core insight, from a 2018 paper by Kraska et al., is almost embarrassingly simple once you hear it: an index is just a function. Give it a key, it gives you back a position. A B-tree approximates this function using pointers and comparisons. But what if you just... learned the function directly, using a small machine learning model? If your keys follow some kind of pattern (and real-world keys almost always do — timestamps, sequential IDs, sorted paths), a simple model can predict "roughly where" a key should be, and you only need a tiny bit of local searching to correct for the model's error.

This project is my attempt to actually build that idea from scratch — no borrowed index libraries, no shortcuts — and see whether it holds up as well as the papers say it does, specifically in the context of filesystem-style metadata lookups.

A quick note on scope, since it matters for how to read the results: this is a study of the index layer only. I'm not building a filesystem. There's no disk I/O, no page cache, no crash recovery. Think of it as isolating one gear in a much bigger machine to see how fast that gear alone can spin.

Some background you'll want before the results make sense

What's wrong with trees, exactly?
Nothing is "wrong" with them — B-trees are one of the great success stories of computer science. But they carry overhead that scales linearly-ish with the number of keys: pointers, node headers, and the branching structure itself all live in memory even though they don't encode any real information about your data, just where to find it. As the number of keys grows into the millions, this overhead adds up and starts to hurt cache performance, since you're bouncing around more distinct memory locations to find one key.

What is a learned index actually doing?
Picture your keys sorted and plotted on a graph: key value on the x-axis, position in the sorted array on the y-axis. If your keys are roughly sequential or evenly distributed, this plot looks a lot like a straight line. A linear regression model can fit that line in almost no space at all — just a slope and an intercept, versus megabytes of tree pointers. Real data is never a perfectly straight line, so the model will be wrong by some small, boundable amount. The trick is to record the worst-case error during training, and then do a tiny binary search within that error window to correct for it. You get most of the win of "just compute the position with math" and none of the risk of silently returning the wrong answer.

The insertion problem.
Here's the catch: a trained model is static. It was fit to a specific dataset. If you insert a new key in the middle of your sorted structure, in principle every prediction downstream of it shifts by one, and your careful error bounds are no longer trustworthy. Trees handle new keys natively (that's what all those pointers are for). Learned indexes need a bolt-on strategy, and this project uses one inspired by the ALEX paper: a small delta buffer — think of it as a scratchpad — where new keys land immediately. Lookups check the main learned model first, and fall back to the delta buffer if they don't find what they're looking for. Periodically, once the buffer fills up, its contents get merged into the main sorted array and the model retrains from scratch on the new, larger dataset.

What I actually built

I implemented three index structures, all in C, all built from first principles rather than pulled off a shelf:

1. Sorted array with binary search — the baseline.
This is the simplest thing that could possibly work. Keep everything sorted, use binary search (O(log n)) to look things up. It needs zero extra memory since the array itself is the index. The catch is insertion: adding a key means shifting every key after it one slot to the right, which is O(n) and gets brutally slow as the array grows. This is precisely the pain point that motivated tree-based indexes in the first place.

2. A block-based B+tree-style index.
I built a simplified two-level structure that looks a lot like the index blocks you'd find in an SSTable (the kind of file format LevelDB or RocksDB use under the hood). Keys are grouped into fixed-size sorted "leaf" blocks of 128 keys each, and a sparse root array keeps track of the maximum key in each leaf. A lookup does a binary search over the (small) root array to find the right leaf, then a second binary search inside that leaf. It's not a fully general, recursively-structured B+tree like you'd find in production, but it captures the same core performance tradeoffs: reasonably fast lookups, and inserts that only need to shift keys locally within one leaf (with an occasional leaf split when a block overflows).

3. A two-stage learned index (a "Recursive Model Index," or RMI) with a delta buffer.
This is the star of the show. It works in two stages, hence "two-stage":
Stage 0 is a single linear model. Its only job is triage: given a key, decide which of several Stage 1 models is responsible for it.
Stage 1 is a collection of independent linear regression models, each specializing in one slice of the key space, each with its own recorded maximum prediction error from training.

A lookup flows through Stage 0, gets routed to the right Stage 1 model, gets a predicted position plus an error bound, and then does a tiny bounded binary search around that prediction to find the exact spot. New inserts go straight into the delta buffer described above, and get folded into the main structure (triggering a full retrain) once that buffer fills up.

(This project is implemented entirely in C, using nothing but the standard math library — no external indexing or ML frameworks. Everything from the linear regression fitting to the binary search bounds was hand-rolled.)

How I tested it

I generated three synthetic key distributions meant to stand in for realistic filesystem access patterns:

  • Sequential keys — monotonically increasing integers, simulating something like an append-only log.
  • Uniform random keys — randomly generated, then sorted; a reasonable stand-in for hashed file paths, where there's no obvious structure for a model to exploit.
  • Lognormal keys — meant to simulate the kind of clustering you actually see in real systems, where files bunch up around certain timestamps or size ranges. (Because lognormal sampling clusters so heavily, I could only generate about 2.86 million distinct values even when aiming for 5 million — a neat little reminder that "just generate more random numbers" doesn't always get you what you want.)

For lookups, I populated each index and fired off 200,000 random point queries, measuring average nanoseconds per lookup, across dataset sizes ranging up to 5 million keys.

For inserts, I built each index using 80% of a dataset, then streamed in the remaining 20% as inserts — importantly, in randomized order, so the baseline array wouldn't get an unfair "best case" from appending purely at the end. I capped insertion tests at 200,000 keys, mostly because the plain array's insertion cost gets so bad at scale that testing it further wasn't informative.

Everything ran single-threaded on a standard Linux container, so take the absolute numbers as relative comparisons between these three structures on identical hardware, not as some universal benchmark of "how fast an index can be."

Results

Lookups get faster, and the gap widens with scale

(uniform key distribution shown; sequential and lognormal followed similar patterns)

At 10,000 keys, the learned index is already more than twice as fast as the plain array. By 5 million keys, it's roughly 3.4x faster than the array and 4x faster than the B+tree, at 123.2 nanoseconds per lookup. This is the central promise of learned indexes playing out in practice: once your model is trained, a lookup is mostly arithmetic (a couple of multiplications and additions) plus a tiny bounded search, instead of a full O(log n) walk through comparisons scattered across memory.

One thing worth flagging honestly: in a few configurations, my hand-rolled B+tree was actually a touch slower than the plain sorted array. I don't think that's a knock against B+trees as a concept — it's more likely an artifact of my leaf size choice (128 keys) causing extra cache misses. A production-grade B+tree implementation, tuned properly, would very likely beat the plain array comfortably. Worth keeping in mind if you go implement one yourself for a class project.

The memory story is the most dramatic part

Across every dataset size tested, the learned index used about 688x less memory than the B+tree just to store the index itself. This isn't a small optimization — it's close to three orders of magnitude. The reason is almost philosophical: a tree stores its structural knowledge (pointers, node boundaries) explicitly, taking up real memory proportional to the number of keys. A learned index stores that same knowledge implicitly, compressed into the coefficients of a handful of linear equations. You're trading explicit bookkeeping for compressed, approximate knowledge plus a cheap correction step — and it turns out that trade is extremely favorable when your data has any regularity at all. Practically, this also means more of your index can fit in CPU cache, which is a big part of why the lookup times above are so much faster too — it's not just fewer instructions, it's fewer cache misses.

Inserts: fast on average, but with a hidden cost

At 200,000 keys: the plain array took 19.1 microseconds per insert (all that shifting adds up), the B+tree took 2.78 microseconds, and the learned index averaged just 1.14 microseconds per insert. On the surface, the learned index wins again. But that average is hiding something important: most inserts are cheap because they just land in the small delta buffer. The expensive part — a full model retrain — happens rarely, but when it happens, it's a real spike in cost. An average is a nice number to put in a table, but for a real system, you'd care a lot about that tail latency too.

Lookups don't degrade much even while you're actively inserting

This was the result that surprised me most. I expected lookup latency to creep upward as the delta buffer filled up (since lookups now have to check two places: the model, then the buffer, if the model misses). Instead, lookup latency stayed close to 117 nanoseconds throughout a whole stream of inserts — still comfortably beating the B+tree's roughly 250 nanoseconds under the same conditions. The delta buffer did its job: it absorbed new writes without meaningfully degrading the accuracy (and therefore search cost) of the underlying model.

What this means, and where it stops meaning that

If you're building something that's mostly read, with data that doesn't change all that often, and the keys have any regular structure (timestamps, sequential IDs, roughly sorted paths — which describes a lot of real-world metadata), a learned index is a genuinely compelling option: faster lookups, and a memory footprint small enough to matter for cache-sensitive systems.

The moment your workload is write-heavy or highly concurrent, the story gets more complicated. The delta buffer strategy used here works, but it's meaningfully more moving parts than "a B+tree splits a node when it's full." You now have two structures to keep consistent, a periodic retrain that needs to happen safely, and — in a real production system — you'd need to think hard about what happens to lookups while a retrain is in progress. That's a genuinely hard concurrency problem, and it's the kind of thing that separates "cool research result" from "thing you'd actually ship."

Where would this actually be useful? A few real candidates: read-heavy metadata layers for object storage or ML training data pipelines (millions of small files, rarely modified after being written), in-memory database indexes where the working set needs to fit in cache, and flash translation layers in SSDs, which map logical to physical addresses — a problem with a strikingly similar shape to this one, and something people have already started exploring in the literature.

Limitations, said plainly

  • This never touched a disk. No I/O, no page cache effects — real systems are usually I/O-bound in ways this study doesn't capture at all.
  • Synthetic data isn't real data. My three distributions are reasonable proxies, but real filesystem traces have bursty, messier patterns that are harder to model cleanly.
  • Single-threaded, no locking. A real metadata layer has to handle many concurrent operations safely — this project sidesteps that problem entirely.
  • The B+tree baseline is simplified. A serious, optimized implementation would likely close some of the performance gap shown here.
  • This is an index, not a filesystem. There's no crash recovery, no actual file storage — just the lookup structure.

Where I'd take this next

The obvious next step is hooking this up to real storage — memory-mapped or direct I/O — to see how much of this advantage survives once disk latency enters the picture. I'd also like to replace the delta buffer with a proper gapped-array design (closer to what ALEX does), which would allow true in-place insertions instead of the buffer-and-retrain approach used here. Beyond that, real concurrency control and testing against actual filesystem traces (rather than synthetic distributions) feel like the two things standing between "a fun from-scratch project" and something you could point to as evidence this approach works in practice.

Related work, if you want to go deeper

  • TableFS and BetrFS — swap in database storage engines (LSM-trees, B-epsilon trees) in place of traditional filesystem metadata structures.
  • The Recursive Model Index (RMI), Kraska et al. — the original paper proposing learned indexes as a replacement for classic tree indexes.
  • FITing-Tree and the PGM-index — extend the idea with formal, provable error bounds.
  • RadixSpline — focuses on making the model-building step itself fast.
  • ALEX — introduces gapped arrays specifically to handle inserts efficiently, which is what inspired the delta buffer design here.
  • SwiftKV — applies learned indexes to distributed filesystem metadata specifically, combining them with LSM-trees.
